# Cross-Agent and Cross-Domain Authority

**Status:** v0.5.0 planning concept note
**AAEF baseline:** v0.4.1 Public Review Draft
**Issue context:** #3

## Purpose

This note defines Cross-Agent and Cross-Domain Authority as a v0.5.0 planning concept for AAEF.

It explains how authority, principal context, delegation constraints, evidence, and revocation semantics should be handled when agentic workflows cross agent, system, organizational, or trust-domain boundaries.

This document does not add new control requirements yet. It is intended to prepare future refinements to Principal Binding, Delegation and Authority, Action Authorization, Evidence, Response, and Trusted Control Boundary guidance.

## Definition

**Cross-Agent and Cross-Domain Authority** refers to the problem of determining whether one agent, system, workflow, or organization may rely on another agent's identity, authority claims, delegated scope, evidence, or execution results.

This includes cases where:

- one agent delegates work to another agent;
- an internal agent calls an external agent;
- an agent crosses an organizational or tenant boundary;
- a workflow relies on an external agent's capability declaration;
- a downstream agent receives principal context or delegated authority;
- evidence from one trust domain is used to authorize, explain, or reconstruct actions in another trust domain.

## Core Principle

Agent-to-agent communication does not imply authority.

The ability to send a message, call an endpoint, invoke an agent, receive a task, or exchange context must not be treated as permission to act.

For high-impact actions, cross-agent and cross-domain workflows should require explicit verification of identity, principal binding, delegated authority, constraints, runtime state, and evidence linkage.

## Why This Matters

Agentic AI systems increasingly operate through:

- sub-agents;
- agent teams;
- external agents;
- plugin or tool ecosystems;
- MCP-style servers;
- cross-tenant workflows;
- outsourced workflows;
- autonomous service agents;
- background orchestration systems;
- agent-to-agent task routing.

In these environments, authority can become unclear.

A receiving agent may know that another agent sent a request, but not whether that agent had authority to delegate the task, whether the original principal authorized the action, whether the delegation was constrained, whether revocation state is current, or whether evidence can be independently verified.

## Authority Failure Patterns

### 1. Communication Treated as Delegation

A system treats an incoming agent message as sufficient authority to act.

Example:

- Agent A sends a task to Agent B.
- Agent B executes a high-impact action because the message looked valid.
- No independent delegation or principal authority check occurs.

### 2. Capability Declaration Treated as Trust

A downstream system relies on an external agent's claimed capabilities, safety level, or policy compliance without verification.

Example:

- An external agent claims it can safely handle production changes.
- The receiving workflow treats that claim as sufficient assurance.

### 3. Principal Context Loss Across Domains

A workflow forwards a task across trust domains but loses the original principal, authority basis, purpose, or constraints.

Example:

- An internal user authorizes a limited action.
- The task is forwarded to an external agent.
- The external agent sees only a service account or agent identity.

### 4. Delegation Constraint Loss

Delegated authority becomes broader, weaker, or ambiguous after crossing a system or organizational boundary.

Example:

- A task delegated for a single customer record becomes a broader dataset operation.
- Time, purpose, amount, tool, or resource constraints are not preserved.

### 5. Evidence Trust Gap

A downstream system relies on evidence generated by another trust domain without knowing whether the evidence is complete, tamper-evident, or independently verifiable.

Example:

- A receiving organization accepts an execution log from an external agent provider without linkage to authorization, dispatch, or principal context.

### 6. Revocation Propagation Gap

Authority is revoked upstream, but downstream agents or external systems continue acting.

Example:

- A principal revokes authority.
- Internal systems stop future actions.
- External delegated agents continue acting from cached authority or stale tasks.

### 7. Cross-Domain Policy Mismatch

Two systems use different meanings for identity, role, authority, approval, evidence, or revocation.

Example:

- One domain treats an approval as workflow-scoped.
- Another domain treats it as reusable for related actions.

## Suggested Authorization Behavior

For high-impact actions, receiving systems should not execute solely because an upstream agent requested an action.

A receiving system should verify:

- upstream agent identity;
- agent instance identity where applicable;
- issuer and operator;
- trust domain;
- principal context;
- authority basis;
- delegated scope;
- delegation constraints;
- expiration or freshness;
- revocation state;
- requested action, resource, purpose, and risk;
- evidence references;
- whether the receiving system's own policy allows the action.

If these cannot be verified, the safer behavior should be denial, deferral, escalation, isolation, freeze, or scoped reauthorization.

## Authority Assertion Considerations

Future AAEF work may define a cross-domain authority assertion profile. A non-normative planning profile is maintained in `docs/en/35-authority-assertion-profile.md`.

Such an assertion could include:

- issuer;
- subject agent;
- agent instance;
- principal reference;
- delegated scope;
- allowed action types;
- resources;
- purpose;
- trust domain;
- expiration;
- delegation depth;
- redelegation permission;
- revocation check reference;
- evidence reference;
- policy version;
- signature or integrity reference where appropriate.

AAEF should not require a specific protocol at this stage.

Possible implementation mechanisms may include signed claims, workload identity, structured delegation records, verifiable credentials, OAuth/OIDC-derived claims, trace context, policy decision artifacts, or domain-specific trust frameworks.

## Evidence Considerations

Future evidence refinements may need to represent:

- upstream agent ID;
- downstream agent ID;
- upstream trust domain;
- downstream trust domain;
- authority assertion reference;
- authority verification result;
- accepted delegated scope;
- rejected delegated scope;
- cross-domain evidence reference;
- revocation check reference;
- receiving-side authorization decision;
- residual trust limitation.

These fields should be generated by trusted workflow, authorization, evidence, or enforcement components. They should not depend solely on model-generated descriptions.

## Assessment Questions

Reviewers may ask:

1. Does the system distinguish communication from delegation?
2. Can external agents be identified, authenticated, and tied to issuers or operators?
3. Can downstream actions be traced back to the original principal and authority basis?
4. Are delegation constraints preserved across agents and domains?
5. Does the receiving system independently authorize high-impact actions?
6. Are cross-domain evidence records independently verifiable or clearly limited?
7. Is revocation or freeze propagation defined for external agents and downstream workflows?
8. Are residual risks recorded when external evidence, authority, or revocation semantics cannot be fully verified?

## Non-Goals

This concept does not require:

- a specific agent-to-agent protocol;
- a specific identity provider;
- a specific credential format;
- a universal cross-domain trust fabric;
- a graph database;
- universal real-time revocation;
- blind trust in external agent claims;
- model self-assessment of cross-domain authority.
