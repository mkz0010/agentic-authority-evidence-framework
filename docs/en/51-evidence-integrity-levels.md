# Evidence Integrity Levels

**Status:** Non-normative v0.5.0 planning model

> **Planning status:** This document is non-normative v0.5.0 planning material. It is not part of the normative v0.4.1 Public Review Draft baseline unless explicitly incorporated into the control catalog, evidence schema, assessment artifacts, testing procedures, or release notes.

## Purpose

This document defines a non-normative planning model for evidence integrity levels in AAEF.

It connects two current v0.5.0 planning themes:

- Tamper-Evident Evidence Storage
- Risk-Proportional Evidence and Performance Overhead

The purpose is to describe how evidence strength can scale with action risk, assurance level, and assessment need without requiring maximum evidence integrity for every agentic action.

This model does not introduce new control requirements by itself.

## Profile Guidance Follow-up

Evidence integrity profile guidance is maintained in `docs/en/60-evidence-integrity-profile-guidance.md`.

That document relates Evidence Integrity Levels to high-impact and audit-grade assessment contexts, evidence depth, tamper-evident evidence, and risk-proportional evidence expectations.

## Design Rationale

AAEF relies on evidence because model output is not authority and model-generated explanations are not sufficient assurance.

However, requiring the strongest possible evidence mechanism for every action would create unnecessary implementation burden, storage cost, latency, and operational complexity.

Evidence integrity should therefore be risk-proportional.

The evidence integrity level should reflect:

- action impact;
- authorization sensitivity;
- reversibility;
- regulatory or contractual expectations;
- audit and incident reconstruction needs;
- whether evidence is independently generated or merely self-reported;
- the trust boundary of the component generating the evidence;
- the likelihood and consequence of evidence tampering.

## Evidence Integrity Level Overview

| Level | Name | Description | Typical use |
| --- | --- | --- | --- |
| EIL-0 | Self-reported evidence | Evidence is produced only by the model, agent runtime, or untrusted execution context. | Debugging, development, or non-assessed exploratory use. |
| EIL-1 | Corroborated runtime evidence | Evidence is generated by the runtime and corroborated by a dispatcher, backend, workflow, or policy component. | Low-to-moderate risk actions where independent corroboration is sufficient. |
| EIL-2 | Centralized evidence writer | Evidence is written by a trusted evidence component or centralized logging path outside the model's direct control. | Assessed actions requiring consistent, action-bound evidence. |
| EIL-3 | Tamper-evident evidence | Evidence is protected by append-only, hash-linked, signed, immutable, externally anchored, or equivalent tamper-evident mechanisms. | High-impact actions, audit-grade review, incident reconstruction, or sensitive authority boundaries. |
| EIL-4 | Strongly isolated evidence plane | Evidence is generated or protected by a strongly isolated control plane, hardware-backed mechanism, managed audit service, or equivalent high-assurance environment. | Highest-impact actions where evidence integrity failure would materially undermine assurance. |

These levels are planning categories, not mandatory implementation prescriptions.

Different systems may satisfy a level through different technical mechanisms if the resulting evidence properties are comparable.

## Level Details

### EIL-0: Self-Reported Evidence

EIL-0 evidence is generated only by the model, agent runtime, or untrusted execution context.

Examples include:

- model-generated explanations;
- agent runtime summaries;
- conversation transcripts without independent corroboration;
- local debug logs controlled by the same component being assessed.

EIL-0 evidence may be useful for debugging or investigation, but should not be treated as sufficient evidence for authority-bearing assessed actions.

### EIL-1: Corroborated Runtime Evidence

EIL-1 evidence is generated by the runtime and corroborated by at least one trusted or semi-trusted system component.

Examples include:

- tool dispatcher logs correlated with agent runtime events;
- backend request logs correlated with authorization decision identifiers;
- workflow system events confirming action attempt and outcome;
- policy component records confirming an allow or deny decision.

EIL-1 is appropriate when the main assurance need is to confirm that an action occurred and was checked by a component outside the model.

### EIL-2: Centralized Evidence Writer

EIL-2 evidence is written through a trusted evidence component or centralized logging path that is outside the model's direct control.

Expected properties include:

- consistent event format;
- action-bound identifiers;
- linkage to authorization context;
- execution outcome capture;
- controlled write path;
- retention suitable for assessment or review.

EIL-2 is a practical baseline for many assessed actions because it reduces dependence on agent runtime self-reporting.

### EIL-3: Tamper-Evident Evidence

EIL-3 evidence provides tamper-evident properties.

Possible mechanisms include:

- append-only evidence records;
- hash chaining;
- signed evidence events;
- immutable storage configuration;
- external anchoring;
- sequence numbers or monotonic event ordering;
- independent evidence replication;
- write-once or restricted modification controls.

EIL-3 does not require one specific technology. The key property is that unauthorized modification, deletion, reordering, or substitution of evidence should be detectable during review.

### EIL-4: Strongly Isolated Evidence Plane

EIL-4 evidence is protected by a strongly isolated evidence plane.

Examples may include:

- hardware-backed attestation or signing;
- isolated control-plane logging;
- managed audit services with strong separation from the agent runtime;
- independently administered evidence storage;
- security-monitoring pipelines with restricted modification capability.

EIL-4 should be reserved for the highest-risk contexts. It should not be assumed as the default requirement for ordinary agentic workflows.

## Risk-Proportional Evidence Selection

Evidence integrity levels should be selected based on risk and assurance need.

| Action or assessment context | Suggested minimum planning level |
| --- | --- |
| Development-only, non-assessed, or exploratory activity | EIL-0 or EIL-1 |
| Low-risk assessed actions | EIL-1 |
| Moderate-risk authority-bearing actions | EIL-2 |
| High-impact actions | EIL-2 or EIL-3 |
| Audit-grade high-impact actions | EIL-3 |
| Highest-impact or strongly regulated authority boundaries | EIL-3 or EIL-4 |

This table is non-normative. It provides planning guidance only.

The final level should be selected based on the action taxonomy, assessment profile, threat model, operational burden, and evidence requirements in effect for the reviewed system.

## Evidence Properties by Level

| Property | EIL-0 | EIL-1 | EIL-2 | EIL-3 | EIL-4 |
| --- | --- | --- | --- | --- | --- |
| Model-independent corroboration | No | Partial | Yes | Yes | Yes |
| Trusted write path | No | Partial | Yes | Yes | Yes |
| Action-bound evidence | Optional | Expected | Expected | Required for assessed use | Required for assessed use |
| Tamper detection | No | Limited | Operational controls | Expected | Strongly expected |
| External or independent protection | No | Optional | Optional | Recommended | Expected |
| Suitable for high-impact audit-grade use | No | Usually no | Sometimes | Usually yes | Yes |

## Performance and Operational Considerations

Higher evidence integrity levels may increase:

- latency;
- storage volume;
- implementation complexity;
- key management requirements;
- log retention costs;
- monitoring and review workload;
- incident response procedures;
- operational dependency on centralized evidence systems.

AAEF planning should avoid treating evidence integrity as an all-or-nothing requirement.

Instead, implementers should define:

- which actions require stronger evidence;
- which actions can use lighter evidence;
- what evidence is required for denial and failure paths;
- what evidence is required for human approval;
- when tamper-evident storage is required;
- how long evidence must be retained;
- how evidence can be queried during assessment or incident review.

## Relationship to Tamper-Evident Evidence Storage

Tamper-evident evidence storage is represented primarily by EIL-3 and above.

This model intentionally avoids requiring EIL-3 for all agentic actions.

A future AAEF release may decide that some profiles, actions, or assessment levels require EIL-3 properties. Until then, this document remains non-normative planning guidance.

## Relationship to Risk-Proportional Evidence

Risk-proportional evidence means that the required evidence strength should correspond to the risk and assurance need of the action.

This model provides a vocabulary for expressing that relationship.

For example:

- ordinary low-risk actions may need corroborated runtime evidence;
- high-impact actions may need centralized evidence writing and tamper-evident protection;
- audit-grade actions may need stronger integrity properties and evidence retention expectations.
