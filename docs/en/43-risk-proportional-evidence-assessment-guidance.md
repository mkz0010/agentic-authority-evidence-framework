# Risk-Proportional Evidence Assessment Guidance

**Status:** v0.5.0 planning guidance
**AAEF baseline:** v0.4.1 Public Review Draft
**Scope:** Non-normative assessment guidance for evidence depth, assurance value, and operational overhead

## Purpose

This document provides non-normative assessment guidance for applying risk-proportional evidence depth in AAEF v0.5.0 planning.

It explains how assessors can reason about evidence depth without requiring every agentic action to produce audit-grade or tamper-evident evidence.

This guidance connects:

- risk-proportional evidence depth;
- tamper-evident evidence storage;
- approval evidence;
- non-execution and reauthorization evidence;
- evidence minimization;
- assessment review expectations.

This document does not change the Evidence Event Schema, control catalog, testing procedures, or assessment worksheet.

## Core Principle

Evidence depth should scale with action risk.

Low-risk actions should not require the same evidence depth as critical, irreversible, externally impactful, delegated, cross-domain, reauthorization-required, or incident-relevant actions.

At the same time, high-impact actions should not rely only on weak operational logs, model self-reporting, or mutable agent-runtime records.

## Evidence Depth Levels

The following evidence depth levels are planning categories.

| Level | Name | Assessment Meaning |
| --- | --- | --- |
| E0 | No AAEF evidence expected | The action is outside AAEF evidence expectations. |
| E1 | Minimal operational evidence | Evidence supports debugging or routine operational review. |
| E2 | Standard action evidence | Evidence supports basic reconstruction of a normal action. |
| E3 | High-impact evidence | Evidence supports high-impact action reconstruction and control review. |
| E4 | Audit-grade evidence | Evidence supports formal audit, incident response, and accountability review. |
| E5 | Tamper-evident or independently verifiable evidence | Evidence includes integrity, corroboration, or independent verification properties. |

## Assessment Interpretation

Evidence depth is not only about the number of fields.

Assessors should consider:

- who or what generated the evidence;
- whether the evidence is action-bound;
- whether it is linked to authorization, dispatch, execution, non-execution, approval, or reauthorization;
- whether it is generated by a trusted component;
- whether it can be modified by the component being assessed;
- whether it is independently corroborated;
- whether it supports incident reconstruction;
- whether it avoids unnecessary sensitive data exposure;
- whether its trust limitations are documented.

## E0: No AAEF Evidence Expected

### Typical Use

E0 may apply when:

- the action is outside AAEF scope;
- the action is not agentic;
- the action is purely local and non-impactful;
- the system intentionally excludes the action category from AAEF assessment.

### Assessment Expectation

Assessors should verify that E0 is not being used to hide high-impact or externally impactful actions.

### Finding Patterns

Findings may arise when:

- high-impact actions are mislabeled as E0;
- E0 is used without scope rationale;
- excluded actions can affect production systems, funds, access, customer data, safety, legal obligations, or security controls.

## E1: Minimal Operational Evidence

### Typical Use

E1 may apply to:

- low-risk operational actions;
- reversible actions;
- local non-production actions;
- routine status checks;
- low-impact agent activity where detailed evidence would create excessive overhead.

### Expected Evidence

E1 evidence may include:

- timestamp;
- agent or workflow identifier;
- action category;
- outcome;
- basic error or status information.

### Assessment Expectation

E1 evidence is usually insufficient for high-impact action reconstruction.

### Finding Patterns

Findings may arise when E1 evidence is used for actions that require authorization, approval, non-execution review, or incident reconstruction.

## E2: Standard Action Evidence

### Typical Use

E2 may apply to:

- normal agentic actions with limited impact;
- reversible actions;
- actions within clear delegated scope;
- actions where standard operational reconstruction is sufficient.

### Expected Evidence

E2 evidence should usually include:

- agent identity;
- agent instance or workflow identifier;
- principal;
- requested action;
- target resource;
- timestamp;
- result;
- correlation identifier;
- basic authorization or policy reference where applicable.

### Assessment Expectation

E2 evidence should support basic action reconstruction.

### Finding Patterns

Findings may arise when E2 evidence lacks principal, action, resource, result, or correlation details.

## E3: High-Impact Evidence

### Typical Use

E3 should be considered for high-impact actions, including actions that affect:

- production systems;
- access rights;
- security controls;
- financial movement;
- customer data;
- regulated data;
- external communications;
- business-critical workflows;
- safety or legal obligations.

### Expected Evidence

E3 evidence should usually include:

- canonical action ID or action digest;
- principal context;
- authority basis;
- authorization decision;
- policy version;
- requested action;
- target resource;
- risk classification;
- dispatch record;
- execution or non-execution outcome;
- correlation across relevant components;
- evidence source and trust limitation.

### Assessment Expectation

E3 evidence should support high-impact action reconstruction and control review.

### Finding Patterns

Findings may arise when high-impact actions are evidenced only by generic logs, model output, screenshots without action binding, or uncorrelated agent runtime records.

## E4: Audit-Grade Evidence

### Typical Use

E4 should be considered when actions are:

- critical;
- externally impactful;
- irreversible or difficult to reverse;
- subject to audit, legal, regulatory, contractual, or incident review;
- associated with approval, override, denial, reauthorization, or human intervention;
- part of a sensitive operational workflow.

### Expected Evidence

E4 evidence should usually include:

- complete action lifecycle evidence;
- authorization decision evidence;
- approval evidence where applicable;
- dispatch evidence;
- backend execution evidence;
- non-execution evidence where applicable;
- reauthorization evidence where applicable;
- override evidence where applicable;
- retention policy reference;
- evidence trust limitations;
- sufficient correlation for independent review.

### Assessment Expectation

E4 evidence should support audit and incident reconstruction by reviewers who were not involved in the original action.

### Finding Patterns

Findings may arise when:

- evidence is complete only from the agent perspective;
- approval records are not linked to final execution;
- non-execution is not recorded;
- reauthorization lacks original denial linkage;
- override evidence overwrites rather than supplements the original evidence;
- retention or trust limitations are undocumented.

## E5: Tamper-Evident or Independently Verifiable Evidence

### Typical Use

E5 should be considered for:

- critical actions;
- audit-grade evidence requiring integrity protection;
- incident-relevant evidence;
- cross-domain actions;
- delegated high-impact actions;
- disputed actions;
- evidence used for external review;
- actions where the assessed component may be compromised.

### Expected Evidence

E5 evidence may include:

- append-only or write-restricted storage;
- signed evidence records;
- hash chains;
- Merkle roots;
- external timestamp references;
- immutable storage policy;
- integrity verification results;
- independent corroborating logs;
- trusted runtime or attestation references;
- deletion or redaction records;
- evidence trust limitations.

### Assessment Expectation

E5 evidence should make alteration, deletion, replay, reordering, truncation, selective omission, or inconsistency detectable.

### Finding Patterns

Findings may arise when:

- critical evidence can be silently modified;
- evidence is stored only in the component being assessed;
- evidence integrity cannot be verified;
- non-execution or failure records are selectively omitted;
- old evidence can be replayed for a new action;
- integrity claims are not independently reviewable.

## Suggested Evidence Depth Triggers

The following table is a planning guide, not a normative requirement.

| Trigger | Suggested Evidence Depth |
| --- | --- |
| Low-risk reversible action | E1 or E2 |
| Standard agentic action | E2 |
| High-impact action | E3 |
| Approval-required action | E3 or E4 |
| Reauthorization-required action | E3 or E4 |
| Non-executed high-impact action | E3 or E4 |
| Override or exceptional human intervention | E4 |
| Critical action | E4 or E5 |
| Incident-relevant action | E4 or E5 |
| Cross-domain or delegated high-impact action | E4 or E5 |
| Externally reviewed or disputed action | E5 |

## Assessment Questions

Assessors should ask:

1. What evidence depth was expected for the action?
2. Why was that evidence depth selected?
3. Was the action low-risk, high-impact, critical, delegated, cross-domain, approval-required, reauthorization-required, or incident-relevant?
4. Does the evidence support action reconstruction?
5. Does the evidence support authorization review?
6. Does the evidence support dispatch and execution review?
7. Does the evidence support non-execution or reauthorization review where applicable?
8. Does the evidence include trust limitations?
9. Does the evidence minimize sensitive data while preserving reviewability?
10. Does the evidence require tamper-evident or independently verifiable properties?
11. Can the evidence be verified by someone other than the component being assessed?
12. Would the evidence support incident reconstruction if one evidence source were compromised?

## Evidence Minimization

Higher evidence depth does not require unlimited data collection.

Assessors should prefer evidence that is:

- structured;
- action-bound;
- correlated;
- minimally sufficient;
- redacted where appropriate;
- reference-based where appropriate;
- hash-based where appropriate;
- protected by access control and retention policy.

Raw prompts, full payloads, sensitive records, or personal data should not be retained by default unless necessary and justified.

## Anti-Patterns

Risk-proportional evidence should avoid:

- treating model self-reports as strong evidence;
- assigning all actions to E1 for operational convenience;
- assigning all actions to E5 without regard to cost or privacy;
- using evidence depth as a substitute for authorization;
- using approval records without action binding;
- ignoring non-execution evidence;
- ignoring reauthorization evidence;
- ignoring evidence integrity;
- storing excessive sensitive data to compensate for weak design;
- claiming audit-grade assurance from mutable logs.
