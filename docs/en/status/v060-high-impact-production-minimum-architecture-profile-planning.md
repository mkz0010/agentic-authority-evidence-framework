## Purpose

This document defines the initial planning draft for a High-Impact Production Minimum Architecture Profile.

The purpose is to clarify the minimum architecture expectations that should be considered before high-impact agentic actions are enabled in production.

This document is planning material. It does not change active controls, assessment criteria, or the current control and assessment baseline.

## Why this is needed

AAEF v0.6.0 focuses on Practical Adoption Readiness.

The Operational Responsibility Matrix identifies who owns, operates, reviews, and responds to high-impact agentic action responsibilities. This architecture profile complements that operational model by defining the minimum technical architecture assumptions needed for those responsibilities to be meaningful.

Without a minimum architecture profile, organizations may implement AAEF concepts inconsistently, for example by:

- treating model output as an authorization decision
- allowing agent runtimes to bypass tool dispatch enforcement
- relying on self-reported execution evidence
- failing to bind backend execution to an authorization decision
- storing evidence without integrity, retention, or retrieval assumptions
- operating high-impact actions without freeze, rollback, or incident reconstruction paths

The profile should help implementers, operators, security architects, auditors, and risk owners evaluate whether a production deployment has enough structure to support AAEF-style action assurance.

## Scope

This planning draft covers candidate architecture expectations for production systems that allow AI agents or agentic workflows to initiate, request, recommend, dispatch, or influence high-impact actions.

The draft focuses on:

- Trusted Control Boundary expectations
- authorization decision placement
- tool dispatch enforcement placement
- backend relying-party verification
- evidence generation and integrity
- monitoring and SIEM integration points
- emergency freeze and rollback expectations
- third-party and SaaS boundary considerations
- profile boundaries for high-impact production use

## Non-goals

This planning draft does not:

- mandate a specific vendor, cloud, runtime, policy engine, gateway, or SIEM
- define a full reference implementation
- define a complete SDK
- replace secure software design, platform hardening, or enterprise architecture review
- create a certification scheme
- claim legal or regulatory compliance
- change the current active AAEF baseline

## Architecture principle

For high-impact production use, AAEF should preserve the core separation:

> Model output is not authority.

A production architecture should therefore ensure that:

- model output is treated as input, recommendation, or request context
- authorization decisions are made by a trusted control component
- tool dispatch is mediated by an enforcement point
- backend services verify authority before execution
- evidence is generated by trusted components, not only by the agent runtime
- execution, refusal, non-execution, override, and reauthorization are reconstructable

## Candidate minimum production architecture

The following components form the initial candidate minimum architecture for high-impact production use.

| Component | Candidate role |
| --- | --- |
| Agent Runtime | Produces plans, recommendations, or action requests, but does not independently authorize high-impact actions. |
| Action Request Interface | Normalizes action requests before authorization and dispatch. |
| Authorization Decision Point | Evaluates policy, principal context, action scope, risk tier, approval requirements, and constraints. |
| Policy Store | Stores versioned authorization policy and approval rules. |
| Approval Service | Captures required human or organizational approval where applicable. |
| Tool Dispatch Enforcement Point | Enforces authorization before tool invocation or action dispatch. |
| Backend Relying Party | Verifies that requested execution is bound to a valid authorization decision. |
| Evidence Generation Point | Generates action-bound evidence from trusted enforcement, backend, or policy components. |
| Evidence Store | Retains evidence with integrity, retrieval, retention, and access-control expectations. |
| Monitoring / SIEM Integration | Receives operational signals, evidence events, alerts, and suspicious mismatch indicators. |
| Emergency Control Plane | Supports freeze, disablement, rollback, or policy revocation for unsafe agentic behavior. |

## Candidate high-impact production minimum expectations

The following expectations are planning candidates for the minimum architecture profile.

### 1. Agent runtime is not authoritative

The agent runtime should not be the sole source of authority for high-impact actions.

Candidate expectation:

- Agent output may request, recommend, or prepare an action.
- Agent output should not by itself authorize execution.
- The authorization decision should be produced by a trusted control component outside the model output path.

### 2. High-impact action requests are normalized

High-impact action requests should be represented in a structured form before authorization.

Candidate expectation:

- The request identifies the principal, agent, action type, target resource, requested parameters, and context.
- The request can be hashed, logged, compared, and bound to later authorization and dispatch artifacts.
- The request distinguishes trusted control inputs from untrusted content or model-derived rationale.

### 3. Authorization decision is explicit and bounded

A high-impact action should have an explicit authorization decision before dispatch.

Candidate expectation:

- The decision includes decision ID, issuer, principal, action scope, policy version, decision result, effective time, expiration, and constraints.
- The decision is bound to the requested action.
- The decision is not reusable outside its intended action, principal, resource, time, or context scope.

### 4. Tool dispatch is mediated

High-impact tool invocation should pass through an enforcement point.

Candidate expectation:

- The tool dispatch layer verifies authorization before dispatch.
- Dispatch is denied or fails closed when authorization is missing, expired, malformed, or mismatched.
- Dispatch evidence records the relationship between request, decision, tool call, and result.

### 5. Backend relying parties verify authority

The backend service that executes a high-impact action should not rely only on the agent runtime or tool gateway assertion.

Candidate expectation:

- The backend verifies that the request is bound to a valid authorization decision.
- The backend checks relevant fields such as action type, resource, principal, policy version, expiration, and action hash.
- The backend rejects execution when the authorization artifact is invalid, expired, mismatched, or outside scope.

### 6. Evidence is generated by trusted components

Evidence should not rely solely on agent self-reporting.

Candidate expectation:

- Evidence is generated or corroborated by trusted enforcement, backend, policy, approval, or evidence components.
- Evidence records authorization, dispatch, backend verification, execution result, refusal, override, or non-execution as applicable.
- Evidence is action-bound and sufficient to support reconstruction.

### 7. Evidence integrity and retrieval are planned

Evidence should be retained in a way that supports audit, incident response, and operational review.

Candidate expectation:

- Evidence retention and retrieval responsibilities are assigned.
- Evidence records include enough correlation identifiers to reconstruct the action path.
- Evidence integrity protections are considered for high-impact actions.
- Evidence access is controlled to reduce tampering, overexposure, or privacy risks.

### 8. Monitoring detects missing or inconsistent signals

Production deployment should monitor for evidence and execution inconsistencies.

Candidate expectation:

- Missing evidence, authorization-to-dispatch mismatch, expired authorization use, backend rejection, policy bypass, and abnormal exception patterns are monitorable.
- Alerts are routed to an operational owner or SOC / incident response function.
- Monitoring supports investigation rather than relying only on aggregate metrics.

### 9. Emergency freeze and rollback paths exist

High-impact agentic action capability should be disableable or reducible when safety assumptions fail.

Candidate expectation:

- There is a defined owner for emergency freeze or disablement.
- Policy rollback, agent disablement, tool revocation, or gateway blocking can be performed.
- Freeze and rollback decisions are recorded.

### 10. Third-party responsibility boundaries are explicit

If SaaS agents, external tools, managed services, or third-party gateways are involved, responsibility boundaries should be recorded.

Candidate expectation:

- The organization identifies which components it controls directly and which are externally operated.
- Evidence, logging, retention, and incident response responsibilities are mapped.
- Gaps are visible to risk owners before production use.

## Minimum verification chain

The minimum production architecture should support a verifiable chain from request to evidence.

Candidate chain:

1. Agent or workflow prepares a high-impact action request.
2. The request is normalized and assigned correlation identifiers.
3. Authorization Decision Point evaluates the request against policy.
4. Approval Service is invoked if required.
5. Authorization Decision Artifact is issued, denied, or withheld.
6. Tool Dispatch Enforcement Point verifies the decision before dispatch.
7. Backend Relying Party verifies the decision before execution.
8. Execution, non-execution, refusal, override, or reauthorization is recorded.
9. Evidence is generated by trusted components.
10. Evidence is stored, monitored, and retrievable for review.

## Candidate backend relying-party verification fields

The following fields are candidates for backend verification.

| Field | Verification purpose |
| --- | --- |
| authorization_decision_id | Confirms the action is tied to a specific authorization decision. |
| decision_issuer | Confirms the decision came from an expected trusted component. |
| principal_id | Confirms the principal is the authorized actor or delegated authority. |
| agent_id | Confirms the requesting agent or workflow is in scope. |
| action_type | Confirms the executed action matches the authorized action. |
| resource_id | Confirms the target resource is within authorized scope. |
| action_hash | Confirms the request payload or canonical action representation has not changed. |
| policy_version | Confirms the decision was evaluated under a known policy version. |
| issued_at | Supports freshness and reconstruction. |
| expires_at | Prevents stale authorization reuse. |
| constraints | Confirms limits, parameters, approval requirements, or execution boundaries. |
| dispatch_id | Binds backend execution to the mediated dispatch path. |
| evidence_required | Indicates whether execution must produce or link to required evidence. |

## Trusted Control Boundary planning assumptions

The Trusted Control Boundary should include the minimum components needed to make authorization, enforcement, and evidence trustworthy.

Candidate in-boundary components:

- Authorization Decision Point
- Policy Store
- Approval Service where approval is required
- Tool Dispatch Enforcement Point
- Evidence Generation Point
- Evidence Store integrity controls
- Emergency freeze or disablement controls

Candidate out-of-boundary or partially trusted components:

- model output
- untrusted content sources
- user-provided free text
- external web content
- agent scratchpads or reasoning traces
- third-party tools without relying-party verification
- logs generated only by the agent runtime

This distinction does not mean out-of-boundary components are unused. It means they should not be treated as sufficient authority or sufficient evidence by themselves.

## Architecture placement examples

### MCP-style tool gateway

Candidate placement:

- Agent Runtime sends structured action requests to a gateway.
- Gateway acts as the Tool Dispatch Enforcement Point.
- Authorization Decision Point may be integrated with or separate from the gateway.
- Backend services still verify authorization where feasible.
- Evidence is emitted by gateway and backend components.

### Policy engine integration

Candidate placement:

- Policy engine supports Authorization Decision Point behavior.
- Policy decisions are versioned and tied to action scope.
- Policy version is included in authorization and evidence records.
- Policy changes are subject to operational change management.

### SIEM integration

Candidate placement:

- Evidence events and alert signals are forwarded to SIEM or equivalent monitoring.
- Missing evidence, mismatched authorization, denied high-impact actions, and exceptions are alertable.
- SIEM records should support investigation and reconstruction rather than only dashboarding.

### Backend service integration

Candidate placement:

- Backend service performs relying-party verification.
- Backend rejects execution if authorization is missing, expired, or mismatched.
- Backend emits corroborating evidence for execution or rejection.

## Candidate architecture anti-patterns

The following patterns should be treated as candidates for explicit warning or prohibition in future guidance:

| Anti-pattern | Risk |
| --- | --- |
| Agent-only authorization | Treats model output as authority. |
| Tool gateway accepts natural-language approval only | Approval cannot be reliably bound to execution. |
| Backend executes based only on gateway trust | Backend cannot detect replay, mismatch, or bypass. |
| Evidence generated only by agent logs | Evidence is self-reported by the least trusted component. |
| No expiration on authorization decisions | Stale decisions can be reused. |
| No action binding or action hash | Authorized and executed actions may diverge. |
| No emergency disablement path | Unsafe behavior cannot be quickly contained. |
| No third-party responsibility mapping | SaaS or external provider gaps are hidden from risk owners. |

## Relationship to operational responsibility

This profile depends on the Operational Responsibility Matrix.

Architecture expectations should be mapped to operational owners, including:

- who owns the Authorization Decision Point
- who owns policy updates
- who owns tool dispatch enforcement
- who owns backend verification
- who owns evidence generation and retention
- who owns monitoring and alert response
- who owns emergency freeze or rollback
- who accepts residual architecture risk

A technically sound architecture without assigned operational ownership is not sufficient for practical adoption.

## Candidate maturity levels

The eventual profile may distinguish between maturity levels.

| Level | Candidate meaning |
| --- | --- |
| Planning | Architecture is described, but not validated for production use. |
| Minimum production candidate | Authorization, dispatch enforcement, backend verification, evidence, monitoring, and freeze paths are defined. |
| Hardened production candidate | Integrity, replay resistance, segregation of duties, monitoring, and incident reconstruction are strengthened. |
| Future advanced candidate | Remote attestation, hardware-backed trust, stronger tamper-evidence, or formal verification may be explored. |

These levels are planning candidates only and should not be treated as certification levels.

## Open questions

The following questions should be resolved before promoting this planning draft into an adoption-facing architecture artifact:

- Which components are mandatory for the minimum production profile?
- Should backend relying-party verification be required for all high-impact actions or recommended where feasible?
- Which authorization artifact fields are minimum versus recommended?
- What evidence integrity level is sufficient for v0.6.0 adoption guidance?
- How should SaaS-only deployments represent tool dispatch enforcement and backend verification?
- Should emergency freeze be mandatory for high-impact production use?
- Should policy engine placement be specified or left implementation-neutral?
- Which architecture anti-patterns should be explicitly documented?
- How should this profile align with future implementer quick-start guidance?
