## Purpose

This document defines an initial planning draft for an AAEF Implementer Quick Start.

The purpose is to give implementers a minimal, adoption-facing implementation path for high-impact agentic action assurance.

This document is planning material. It does not change active controls, assessment criteria, schema requirements, or the current control and assessment baseline.

## Intended audience

This planning draft is intended for:

- engineers implementing agentic workflows
- platform teams operating tool gateways or action dispatch layers
- backend service owners implementing relying-party verification
- security architects reviewing enforcement placement
- evidence pipeline owners connecting action events to audit or monitoring systems
- technical leads evaluating what a minimum AAEF-oriented implementation requires

## Core implementation idea

For high-impact agentic actions, the implementation should preserve the core AAEF principle:

> Model output is not authority.

The agent runtime may propose, prepare, or request an action.

A trusted authorization layer should decide whether the action is allowed, denied, requires approval, or requires reauthorization.

A dispatch enforcement layer should mediate tool or action invocation.

A backend relying party should verify that execution is bound to a valid authorization decision.

Trusted components should generate evidence that supports reconstruction.

## Minimal implementation chain

A practical implementer quick start should describe the following minimum chain:

1. Agent or workflow prepares an action request.
2. The request is normalized into a structured action request.
3. The authorization layer evaluates the request against policy.
4. An Authorization Decision Artifact is issued, denied, or withheld.
5. The tool dispatch enforcement point validates the decision before dispatch.
6. The backend relying party verifies authority before execution.
7. Evidence is generated by trusted components.
8. Evidence is stored and correlated for audit, monitoring, and incident reconstruction.
9. Failures, denials, refusals, non-execution, and reauthorization are recorded where required.

## Minimum components

The following components are candidates for a minimum implementation path.

| Component | Implementer responsibility |
| --- | --- |
| Agent Runtime | Produces proposed actions or structured action requests, but does not authorize high-impact execution by itself. |
| Action Request Normalizer | Converts model output, workflow intent, or user requests into a canonical action request. |
| Authorization Decision Point | Evaluates policy and issues an authorization decision. |
| Policy Store | Provides versioned policy used by the authorization layer. |
| Approval Service | Captures required approval when policy requires human or organizational approval. |
| Tool Dispatch Enforcement Point | Validates authorization before invoking a tool or action. |
| Backend Relying Party | Verifies authorization before performing the high-impact action. |
| Evidence Generator | Emits action-bound evidence from trusted enforcement, backend, policy, approval, or evidence components. |
| Evidence Store | Retains evidence for retrieval, audit, monitoring, and incident reconstruction. |
| Monitoring Integration | Receives signals for missing evidence, mismatches, denial patterns, and operational failures. |

## Minimal action request shape

The quick start should include a minimal action request shape, separate from the Authorization Decision Artifact.

Candidate fields:

| Field | Purpose |
| --- | --- |
| `request_id` | Unique identifier for the normalized action request. |
| `correlation_id` | Cross-system identifier linking request, decision, dispatch, backend execution, and evidence. |
| `principal_id` | Principal on whose behalf the action is requested. |
| `agent_id` | Agent, workflow, or runtime requesting the action. |
| `action_type` | Type of requested action. |
| `resource_id` | Target resource, record, account, system, or environment. |
| `requested_parameters` | Structured parameters needed for authorization and execution. |
| `request_context` | Trusted context used for authorization. |
| `untrusted_input_summary` | Optional summary or reference to untrusted content influence, without treating it as authority. |
| `requested_at` | Time the action request was created. |

## Minimal authorization decision usage

The quick start should refer to the Authorization Decision Artifact Minimal Profile rather than redefining it.

At minimum, implementers should be able to:

- submit a normalized action request to an authorization decision component
- receive an explicit decision result
- bind the decision to the requested action
- enforce expiration and constraints
- pass decision identifiers to dispatch and backend verification
- link the decision to evidence records

Candidate decision results:

| Result | Implementation behavior |
| --- | --- |
| `allow` | Dispatch may proceed if constraints and time bounds are satisfied. |
| `deny` | Dispatch must not proceed; denial evidence may be generated. |
| `approval_required` | Dispatch must wait for valid approval. |
| `reauthorization_required` | A fresh decision must be obtained before dispatch. |
| `not_applicable` | The request should not be treated as authorized by this decision point. |

## Minimal dispatch enforcement behavior

Before dispatching a high-impact action, the enforcement point should verify:

- an authorization decision is present when required
- the decision issuer is trusted
- the decision result allows dispatch
- the decision has not expired
- the action type matches the requested tool or operation
- the resource is within scope
- the principal and agent are in scope
- the action hash or equivalent binding matches the normalized request
- constraints are satisfied
- required approval has been captured where applicable
- evidence requirements are known before execution

If validation fails, dispatch should fail closed.

## Minimal backend relying-party behavior

The backend relying party should not treat the agent runtime or gateway assertion as sufficient authority.

Before executing a high-impact action, the backend should verify:

- the execution request is bound to a valid authorization decision
- the decision has not expired
- the action type and resource match the decision scope
- the principal and agent are expected
- constraints are still satisfied
- the dispatch path is expected
- execution can generate or link required evidence

If verification fails, the backend should reject execution and generate rejection or non-execution evidence where required.

## Minimal evidence behavior

Evidence should support reconstruction of:

1. the normalized action request
2. the authorization decision
3. approval where applicable
4. dispatch enforcement
5. backend verification
6. execution, rejection, refusal, non-execution, override, or reauthorization
7. policy version and relevant constraints
8. evidence generation point and timestamp

The quick start should emphasize that agent self-reporting is not sufficient evidence by itself for high-impact production use.

## Minimal implementation sequence

A practical quick start should include an implementation sequence such as:

1. Identify high-impact actions.
2. Place those actions behind a dispatch enforcement point.
3. Define a normalized action request shape.
4. Implement an authorization decision component or adapter.
5. Bind authorization decisions to action requests.
6. Implement dispatch validation.
7. Implement backend relying-party verification where feasible.
8. Generate trusted evidence events.
9. Route evidence and alerts to monitoring or SIEM where applicable.
10. Define failure, denial, non-execution, and reauthorization behavior.
11. Assign operational owners using the Operational Responsibility Matrix.
12. Review architecture placement using the High-Impact Production Minimum Architecture Profile.

## Minimal implementation checklist

Before an implementation is considered ready for high-impact pilot use, the implementer should be able to answer:

- What high-impact actions are in scope?
- Where are action requests normalized?
- Which component issues authorization decisions?
- Which policy version is used?
- Where is dispatch enforcement performed?
- What happens when authorization is missing, expired, or mismatched?
- Which backend services verify authority before execution?
- Which trusted components generate evidence?
- How are request, decision, dispatch, backend execution, and evidence correlated?
- How are denials, refusals, and non-execution recorded?
- Who receives alerts when evidence is missing or inconsistent?
- Who can freeze or disable high-impact execution?
- Which responsibilities map to the Operational Responsibility Matrix?

## Candidate implementation modes

The quick start should remain implementation-neutral, but may describe common patterns.

| Mode | Candidate description |
| --- | --- |
| Gateway-mediated | A tool gateway mediates high-impact action dispatch and performs decision validation. |
| Backend-enforced | Backend services perform strong relying-party verification before execution. |
| Policy-engine-backed | A policy engine issues or supports authorization decisions. |
| SaaS-constrained | Some enforcement or evidence capabilities depend on third-party or SaaS provider features. |
| Hybrid | Gateway, backend, policy, and evidence responsibilities are split across internal and external components. |

## Failure and non-execution behavior

The quick start should explain that failures are part of the assurance model.

Candidate failure cases:

| Failure case | Expected behavior |
| --- | --- |
| Missing authorization decision | Dispatch denied or blocked. |
| Expired decision | Dispatch or backend execution rejected. |
| Action mismatch | Dispatch or backend execution rejected. |
| Missing approval | Dispatch held or denied. |
| Backend verification failure | Execution rejected. |
| Evidence generation failure | Execution may be blocked, flagged, or escalated depending on policy. |
| Policy unavailable | Fail closed for high-impact actions unless an explicit emergency process exists. |
| Monitoring unavailable | Escalate or restrict high-impact execution depending on risk policy. |

## Candidate anti-patterns

| Anti-pattern | Risk |
| --- | --- |
| Direct model-to-tool execution for high-impact actions | Model output effectively becomes authority. |
| No normalized action request | Authorization and execution cannot be reliably compared. |
| Authorization decision not bound to action | Dispatch or backend execution may diverge from authorization. |
| Gateway-only trust without backend verification | Backend cannot detect bypass, replay, or mismatch. |
| Agent-only evidence | Evidence is self-reported by the least trusted component. |
| No denial or non-execution record | Failures cannot be reviewed or audited. |
| No owner for policy or evidence | Implementation cannot be operated or assessed reliably. |
| No freeze path | Unsafe behavior cannot be contained quickly. |
